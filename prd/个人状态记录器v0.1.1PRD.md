# 《个人状态记录器 v0.1.1 PRD》

版本：v0.1.1  
产品名：个人状态记录器  
版本定位：**手机可用版 / 数据迁移版**  
基于版本：v0.1  
目标平台：移动端优先 PWA / Web App  
核心目标：让 v0.1 能够在手机上长期使用，并支持电脑与手机之间的数据迁移。

---

# 1. 版本背景

v0.1 已经基本完成，当前主要问题是：

> 应用目前是网页版，本地运行时只能在电脑上使用，离开电脑后无法继续记录。

由于「个人状态记录器」的核心使用场景是每日随手记录，因此手机端使用非常重要。

v0.1.1 不做功能扩展，不增加复盘、标签、日历、AI 等能力，只解决三个基础问题：

1. 手机可以访问；
2. 手机可以像 App 一样添加到桌面；
3. 电脑已有数据可以迁移到手机。

---

# 2. v0.1.1 核心目标

## 2.1 产品目标

让「个人状态记录器」从本地网页版变成一个可以在手机上长期使用的轻量 PWA。

## 2.2 功能目标

v0.1.1 只做以下内容：

1. 支持 PWA；
2. 支持部署到 Vercel / Netlify / Cloudflare Pages；
3. 支持手机添加到主屏幕；
4. 保持 LocalStorage 本地存储；
5. 增加 JSON 导入功能；
6. 优化数据导出/导入说明；
7. 保证现有 v0.1 功能不受影响。

---

# 3. v0.1.1 不做事项

本版本明确不做：

1. 不做登录注册；
2. 不做云同步；
3. 不做后端服务；
4. 不做数据库；
5. 不做 AI 分析；
6. 不做周复盘；
7. 不做标签系统；
8. 不做日历视图；
9. 不做提醒推送；
10. 不做多设备自动同步；
11. 不做应用商店上架；
12. 不做微信小程序版本。

本版本原则：

> 只解决“手机能不能方便使用”和“数据能不能迁移”。

# 4. 用户场景

## 场景 1：手机打开应用

用户通过部署后的公网地址访问应用：

```text
https://xxx.vercel.app
```

在手机浏览器中打开后，可以正常使用所有 v0.1 功能。

---

## 场景 2：添加到手机主屏幕

用户在手机浏览器中选择：

- iPhone：Safari → 分享 → 添加到主屏幕；
- Android：Chrome → 添加到主屏幕 / 安装应用。

添加后，桌面出现「个人状态记录器」图标。

用户点击图标后，应用以接近原生 App 的形式打开。

---

## 场景 3：电脑数据迁移到手机

用户之前在电脑浏览器中记录了一些数据。

迁移流程：

1. 在电脑端设置页点击「导出 JSON」；
2. 将 JSON 文件发送到手机；
3. 手机端打开应用；
4. 设置页点击「导入 JSON」；
5. 选择文件；
6. 系统读取并合并数据；
7. 手机端继续记录。

---

## 场景 4：手机数据备份

用户在手机端使用一段时间后，可以定期导出 JSON 备份。

# 5.3 JSON 导入功能

## 5.3.1 功能描述

v0.1 已支持导出 JSON。  
v0.1.1 增加导入 JSON，用于设备间迁移数据。

入口位置：

```text
设置页
```

新增按钮或文件选择区域：

```text
导入 JSON
```

---

## 5.3.2 导入流程

用户点击「导入 JSON」后：

1. 打开文件选择器；
2. 用户选择 `.json` 文件；
3. 系统读取文件内容；
4. 解析 JSON；
5. 校验数据格式；
6. 如果合法，与本地数据合并；
7. 保存到 LocalStorage；
8. 显示导入成功提示。

---

## 5.3.3 合法 JSON 格式

JSON 应为数组：

```ts
StateRecord[]
```

示例：

```json
[
  {
    "id": "2026-05-15",
    "date": "2026-05-15",
    "energy": 6,
    "spark": 3,
    "action": 5,
    "connection": 2,
    "expression": 3,
    "stability": 6,
    "positiveNote": "今天看了一部有意思的电影",
    "drainNote": "工作内容重复，有点无聊",
    "actionNote": "整理了房间",
    "createdAt": "2026-05-15T21:30:00",
    "updatedAt": "2026-05-15T21:30:00"
  }
]
```

---

## 5.3.4 基础校验规则

导入数据必须满足：

### 1. 顶层必须是数组

如果不是数组，提示：

```text
导入失败：文件格式不正确。
```

---

### 2. 每条记录必须包含以下字段

```text
date
energy
spark
action
connection
expression
stability
```

---

### 3. 分值字段必须是 number

字段：

```text
energy
spark
action
connection
expression
stability
```

必须是数字。

---

### 4. 分值必须在 0-10 之间

如果有分值不在范围内，视为非法。

---

### 5. date 必须是字符串

格式建议为：

```text
YYYY-MM-DD
```

v0.1.1 可做简单校验：

```ts
/^\d{4}-\d{2}-\d{2}$/
```

---

## 5.3.5 导入合并策略

采用简单覆盖策略。

### 规则

1. 以 `date` 作为唯一标识；
2. 如果本地已有同日期记录，则用导入记录覆盖本地记录；
3. 如果本地没有该日期记录，则新增；
4. 如果导入记录没有 `id`，则自动使用 `date` 作为 `id`；
5. 如果导入记录没有 `createdAt`，则自动补当前时间；
6. 导入时更新 `updatedAt`，或保留导入记录中的 `updatedAt`。

推荐策略：

```text
同日期导入数据覆盖本地数据。
```

原因：简单、可预期、实现成本低。

---

## 5.3.6 导入成功提示

导入成功后显示：

```text
导入成功，共导入 X 条记录，覆盖 Y 条，新增 Z 条。
```

---

## 5.3.7 导入失败处理

如果 JSON 解析失败：

```text
导入失败：无法解析 JSON 文件。
```

如果数据格式不合法：

```text
导入失败：数据格式不符合要求。
```

如果读取文件失败：

```text
导入失败：文件读取失败。
```

重要要求：

> 导入失败时，不能修改原有本地数据。

---

## 5.3.8 验收标准

- 能选择本地 JSON 文件；
- 能成功导入合法 StateRecord[]；
- 非法 JSON 不会破坏现有数据；
- 同日期记录能覆盖；
- 新日期记录能新增；
- 导入后历史记录、趋势图、面板能正常读取新数据；
- 导入成功后有提示。
